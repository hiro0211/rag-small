# 会話コンテキスト管理の改善 — Claude Code向け指示プロンプト

## 背景・目的

このRAGチャットアプリでは、会話が長くなってもLLMが過去の文脈を保持できるように、会話履歴の管理を改善したい。現状の実装には以下のギャップがある。

---

## ギャップ分析（現状 → 理想）

### ギャップ1: `trim_messages` のトークンカウントが文字数ベースになっている

- **対象ファイル**: `lib/graph.py` 67〜74行目
- **現状**: `token_counter=len` で Python の `len()` 関数（＝文字数）をトークン数として使っている。`max_tokens=20` なので、実質20文字＝直近1〜2メッセージしかLLMに送られていない。
- **理想**: `tiktoken` 等の正確なトークンカウンターを使い、`max_tokens` をモデルのコンテキストウィンドウに見合った値（数千〜数万トークン）に設定する。
- **修正方針**:
  - `tiktoken` を `requirements.txt` に追加する。
  - モデルごとに適切なトークンカウンターを返すヘルパー関数を `lib/llm.py` に追加する（gpt-4o-mini は tiktoken の `cl100k_base`、gemini-2.5-flash は文字数ベースの概算でもよいが、1文字≒1.5トークン程度の換算関数を用意する）。
  - `lib/graph.py` の `generate()` 関数で、`token_counter` にそのヘルパー関数を渡す。
  - `max_tokens` をモデルのコンテキストウィンドウの約70%程度に設定する（システムプロンプト＋RAGコンテキスト＋出力分の余裕を残すため）。gpt-4o-mini なら約80,000トークン、gemini-2.5-flash なら約700,000トークン程度を目安とする。
  - `max_tokens` の値はモデルIDに応じて動的に切り替える。ハードコードせず、モデル設定を辞書等で管理することが望ましい。

### ギャップ2: テストコードが現状の不正確なトリミングを前提としている

- **対象ファイル**: `tests/test_graph.py` 113〜139行目（`test_trims_messages_to_last_20`）
- **現状**: `len(non_system) <= 20` というアサーションで、文字数20を前提とした検証になっている。
- **理想**: トークンベースのトリミングに合わせてテストも更新する。
- **修正方針**:
  - テスト名を `test_trims_messages_within_token_limit` 等に変更する。
  - mock で token_counter を差し替えるか、十分な量のメッセージを作成してトリミングが正しく機能することを検証する。
  - トリミング後のメッセージ合計トークン数が `max_tokens` 以下であることをアサートする。

### ギャップ3: Supabaseからの会話履歴取得に上限がない

- **対象ファイル**: `lib/chat_history.py` 28〜38行目（`get_messages`）
- **現状**: `get_messages(session_id)` は全メッセージを取得する。会話が数百ラリーになるとDB取得が遅くなる。
- **理想**: 取得件数に上限を設け、UIの表示とLLMへの送信で必要十分な量だけ取得する。
- **修正方針**:
  - `get_messages()` に `limit` パラメータ（デフォルト100程度）を追加する。
  - 最新のメッセージから逆順で取得し、Python側で時系列順に並び替える。
  - `app.py` 側で呼び出す際にもこの `limit` を利用する。
  - 必要に応じて「もっと読み込む」的なページネーションは将来対応として、今回はスコープ外とする。

### ギャップ4: モデル設定がコード内に散在しており、コンテキストウィンドウ情報がない

- **対象ファイル**: `lib/llm.py`
- **現状**: `create_llm()` はモデルインスタンスを返すだけで、コンテキストウィンドウサイズなどのメタ情報を持っていない。
- **理想**: モデルごとのコンテキストウィンドウサイズ・トークンカウンター情報を一元管理する。
- **修正方針**:
  - `MODEL_CONFIG` のような辞書を `lib/llm.py` に定義する。各モデルについて `context_window`（トークン数）、`token_counter_type`（"tiktoken" or "char_estimate"）、`encoding_name`（tiktoken用）等を保持する。
  - `generate()` から参照できるように `get_model_config(model_id)` のようなヘルパーを用意する。
  - `lib/graph.py` の `generate()` 関数内で、この設定を使って `max_tokens` と `token_counter` を動的に決定する。

---

## 制約・注意事項

- **既存の動作を壊さないこと**: rewrite_query ノードと retrieve ノードはこの修正の影響を受けないはずだが、念のためテストを通すこと。
- **ストリーミング動作の維持**: `stream_response_with_sources` のマルチモードストリーミングの挙動は変えないこと。
- **Gemini のトークンカウント**: Gemini には tiktoken が使えないため、文字数ベースの概算カウンターで代用してよい。ただし、将来的に `google-generativeai` の `count_tokens` API に切り替えられる設計にしておくこと。
- **テストは全て通ること**: 修正後に `pytest tests/` を実行し、全テストがパスすることを確認すること。

---

## 完了基準

1. `lib/graph.py` の `generate()` で tiktoken ベースの正確なトークンカウントが使われている。
2. `max_tokens` がモデルのコンテキストウィンドウに応じた適切な値に設定されている。
3. `lib/llm.py` にモデルのメタ情報（コンテキストウィンドウサイズ等）が一元管理されている。
4. `lib/chat_history.py` の `get_messages()` に取得件数の上限が設定されている。
5. `tests/test_graph.py` のトリミング関連テストが新しい実装に合わせて更新されている。
6. `pytest tests/` が全てパスする。
7. `requirements.txt` に `tiktoken` が追加されている。
